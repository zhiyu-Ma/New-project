# Graphiti 文字冒险 Demo 方案

## 目标

验证 Graphiti 在动态叙事场景下的能力。**不是做游戏，是测试知识图谱链路。**

具体要验证四件事：

1. `add_episode` 从自然语言抽取的图谱质量是否够用
2. 图谱检索返回的上下文是否真能驱动 LLM 生成连贯叙事
3. 时序边（关系随时间演变）在多回合后是否体现价值
4. "信息逐步揭露"这个文字冒险核心机制能否靠图谱 + 自然语言更新自动实现

---

## 整体设定

- 5 个 NPC，1 个小镇
- 纯涌现式叙事，无预设主线
- 菜单驱动，玩家从结构化选项中选择动作
- 初始化与运行时复用同一条链路（都走 `add_episode`）

---

## Graphiti 架构补充

理解这个分层有助于设计和验证。

**Episode 层（原始记忆）**
- `EpisodicNode`：你每次 `add_episode` 输入的原文
- `EpisodicEdge`：连接 EpisodicNode 到它抽出的 EntityNode（溯源链路）

**Entity 层（结构化知识）**
- `EntityNode`：抽出的实体（Person / Location / Topic）
- `EntityEdge`：实体之间的关系，**时序信息（`valid_at` / `invalid_at`）就在这里**

**Community 层（聚类）：** Demo 不用，5 个 NPC 聚类没意义。

你直接打交道的 API 就两个：`add_episode()` 和 `search()`，底下分层自己管。

**一次 `add_episode` 的代价：** 内部 2-3 次 LLM 调用（实体抽取、关系抽取、冲突判定），10 回合下来仅图谱更新就 30+ 次调用，要预估成本。

## 图谱 Schema

### 节点类型

- `Person`：name, personality（2-3 个标签）, current_mood
- `Location`：name, description
- `Topic`：事件/传闻/秘密/物品的统一抽象，"可被谈论的东西"
  - `topic_type`：`event` / `secret` / `rumor` / `item`
  - `truth_value`：`true` / `false` / `unknown`

### 关键设计：秘密作为独立 Topic 节点

不作为 Person 属性。作为属性是死数据，无法被检索、关联、演变。作为节点后：

- 可以表达"谁知道这个秘密"（`KNOWS_ABOUT` 边）
- 可以关联到事件（`INVOLVED_IN` 边）
- 秘密本身可以演化（从私密到半公开）
- 进入 Graphiti 语义检索范围，玩法立刻浮现

**不拆成多个节点类型**（不要单独建 Secret 类、Rumor 类）。Graphiti 的实体抽取对自定义类型支持有限，类型越多 LLM 抽得越乱。统一用 Topic + `topic_type` 属性区分。

### 边类型（建议在 prompt 中提示优先识别）

- `KNOWS` — A 认识 B
- `TRUSTS` / `DISTRUSTS` — 带强度
- `LOCATED_AT` — 人在哪
- `KNOWS_ABOUT` — A 接触过某 Topic（决定菜单可聊话题）
- `INVOLVED_IN` — 人物卷入事件

**Demo 阶段简化：** 不区分"知道"和"相信"。即使是谣言，靠 Topic 节点的 `truth_value` 属性体现真伪，不靠边类型区分。如果跑通后发现剧情扁平，再升级为 `KNOWS_ABOUT` / `BELIEVES` / `SUSPECTS` 三种边。

Graphiti 自动给每条边打时间戳，关系演变原生支持。

## 行为系统

**颗粒度：动词 + 对象 + 可选修饰**

| 动词 | 参数 1 | 参数 2 |
|------|--------|--------|
| 对话 | NPC | 话题（动态） |
| 观察 | NPC 或 地点 | — |
| 打听 | NPC | 打听对象（另一 NPC） |
| 赠送 | NPC | 物品 |
| 前往 | 地点 | — |
| 等待/休息 | — | — |

**关键点：话题列表动态生成。** 只显示主角"知道"的事，从图谱中查询 `主角 -[KNOWS_ABOUT]-> Topic` 得到。这让图谱直接驱动 UI，检索本身成为玩法的一部分。

---

## 单回合链路总览

```
1. 玩家选动作（如：打听-铁匠-财务状况）

2. 双重检索图谱
   ┌─ 主角视角检索：主角已知的人、地、Topic
   │  → 用于生成叙述的"明面信息"
   │
   └─ 上帝视角检索：当前场景涉及的所有 Topic（包括隐藏秘密）
      → 用于决定剧情走向（是否揭露秘密）

3. LLM 调用 1：叙述生成
   输入：场景 + 双重检索结果 + 玩家动作
   输出：JSON（决策 + 100-200 字事件文本）

4. 显示叙述给玩家

5. LLM 调用 2：图谱更新
   → graphiti.add_episode(narrative_text, reference_time=game_time)
   揭露机制：如果叙述中揭露了秘密，新的 KNOWS_ABOUT 边会自动建出
```

**核心机制：揭露不需要写代码，靠叙述文本驱动。** 叙述里写"主角注意到铁匠桌上有催债字条"，Graphiti 抽取时就会建出 `主角 -[KNOWS_ABOUT]-> 高利贷秘密` 这条边。这是 demo 要验证的核心能力。

**关于 `reference_time`：** 必传。维护游戏内递增时间戳传进去，否则时序边都用系统时间，时序逻辑会乱。

**关于 source 类型：**
- 叙述文本用 `EpisodeType.text`
- 玩家动作用 `EpisodeType.json`（结构化记录"做了什么"）

## 模块一：初始化

### 三个常见翻车点

**翻车点 1：NPC 各自独立。** LLM 默认给你 5 个互不相识的人，结果只有"主角 ↔ NPC"的星形结构，不像小镇。

**翻车点 2：秘密之间无关。** 默认输出"独立秘密"——铁匠欠债、寡妇有私生子、镇长贪污，三件事不相干。好的小镇剧情里秘密应该互相牵连，揭露一个会带出别的。

**翻车点 3：信息分布不合理。** 知情者全是当事人 → 主角永远打听不出来；知情者太多 → 不是秘密。

### 初始化 Prompt 模板

````
# 任务

你要为一个文字冒险游戏生成一个小镇的初始世界状态。这个世界将作为知识图谱的种子数据，后续会随玩家行为演化。

# 设定

- 小镇规模：1 个小镇，约 5 个关键 NPC
- 风格：现实向，可以有戏剧冲突，但不要奇幻、超自然元素
- 时代：[填入想要的时代，如"民国小镇"、"现代北方县城"]
- 时间锚点：故事开始前的"过去一年"内已发生若干事件

# 必须包含的元素

## NPC（5 个）
每人需要：
- 全名（后续叙述中将统一使用此名，不要使用别名或职业代称）
- 职业 / 身份
- 2-3 个性格标签
- 当前状态（情绪基线 + 一句话现状）
- 一个具体的"想要的东西"或"害怕的东西"

## 地点（3-5 个）
小镇内的关键场所，每个地点要标注谁常在此出没。

## 公开事件（2-3 个）
镇上人尽皆知的近期事件。这些事件要和至少一个 NPC 有牵连。

## 秘密（4-5 个）
每个秘密包含：
- 一句话描述
- 知情者列表（1-3 人）
- 真实性（true / false / 半真半假）
- 严重程度

**关键约束（必须满足）：**

1. 至少 2 个秘密涉及 2 个以上 NPC（不要全是单人秘密）
2. 至少 1 个秘密与某个公开事件相关（公开事件背后的真相）
3. 至少 1 处信息不对称：A 知道关于 B 的某事，但 B 不知道 A 知道
4. 秘密之间至少有 1 处交叉（一个秘密的揭露会牵出另一个秘密）

## NPC 之间的关系（至少 6 条）

注意：是 NPC 互相之间的关系，不是 NPC 和主角的关系。
关系类型例如：信任、敌意、欠人情、暗恋、生意伙伴、远亲等。

每条关系标注：
- 双方
- 关系性质
- 简短由来（一句话）

## 主角与世界的初始连接

- 主角的身份（外来者？本地人？返乡？）
- 主角认识哪些 NPC，认识程度如何
- 主角已知哪些公开事件
- 主角是否已知任何秘密（建议初始化时主角不知道任何秘密）

# 输出格式

严格按以下 JSON 输出，不要额外说明：

{
  "setting": {
    "town_name": "...",
    "era": "...",
    "atmosphere": "一句话氛围描述"
  },
  "npcs": [
    {
      "id": "npc_01",
      "full_name": "...",
      "occupation": "...",
      "personality_tags": ["...", "..."],
      "current_state": "...",
      "motivation": "..."
    }
  ],
  "locations": [
    {
      "id": "loc_01",
      "name": "...",
      "description": "...",
      "regulars": ["npc_01", "npc_03"]
    }
  ],
  "public_events": [
    {
      "id": "event_01",
      "description": "...",
      "involved_npcs": ["npc_02"],
      "when": "上周 / 三个月前 / 等"
    }
  ],
  "secrets": [
    {
      "id": "secret_01",
      "description": "...",
      "knowers": ["npc_01"],
      "truth_value": "true",
      "severity": "...",
      "related_to": ["event_01", "secret_03"]
    }
  ],
  "npc_relationships": [
    {
      "from": "npc_01",
      "to": "npc_02",
      "type": "distrust",
      "origin": "..."
    }
  ],
  "protagonist": {
    "background": "...",
    "knows_npcs": [
      {"npc_id": "npc_01", "familiarity": "熟人 / 点头之交 / 陌生"}
    ],
    "knows_events": ["event_01"],
    "knows_secrets": []
  }
}
````

### 设计要点

**为什么用 ID 系统：** 让 LLM 在不同段落引用同一实体时强制一致。否则它会在 `secrets` 里写"铁匠"、在 `relationships` 里写"张铁匠"——你要做大量字符串清理。

**为什么先输出 JSON 再转 episode：** 直觉上让 LLM 直接生成自然语言更直接。但 JSON 输出可以 schema 校验、可以控制每条 episode 的颗粒度（一条 episode 讲一件事，避免 Graphiti 抽取混乱）、信息可重复利用。

### JSON → episodes 转换规则

JSON 拿到后，按下面规则拆成多条 episode 喂给 Graphiti：

```
For each NPC:
    episode: "[全名]是[职业]。性格[标签]。[现状]。"

For each location:
    episode: "[地点名]：[描述]。常出现在这里的人：[regulars 名字]。"

For each public_event:
    episode: "[when]，[描述]。涉及[involved_npcs 名字]。"

For each secret:
    episode: "[knowers 名字]知道一件事：[描述]。"
    (作为"X 知道 Y"的事实写入，Graphiti 会建出 KNOWS_ABOUT 边)

For each npc_relationship:
    episode: "[from 名字]对[to 名字]的态度：[type]。原因：[origin]。"

For protagonist:
    episode: "主角是[background]。"
    For each knows_npcs:
        episode: "主角认识[名字]，[familiarity]。"
```

**关键细节：** 每条 episode 用全名，不用 ID。Graphiti 不认 ID，认的是文本里的实体名字。ID 只在你代码内部做引用一致性。

### 初始化质量验证

JSON 生成后，跑 demo 之前先检查硬指标：

```
NPC 数量 == 5
NPC 间关系数量 >= 6
秘密数量 >= 4
多人秘密数量 >= 2
秘密-事件关联数量 >= 1
秘密之间关联数量 >= 1
```

不达标就让 LLM 重新生成。**初始化质量直接决定后续 10 回合的剧情上限**——种子图谱平淡，后面再怎么跑也起不来。这一步多花点钱重试是值得的。

### 可选：戏剧张力强化

如果生成的世界总是"安全无聊"，在 prompt 末尾加：

```
# 戏剧张力要求
至少要有：
- 1 个 NPC 的"想要的东西"和另一个 NPC 的"想要的东西"直接冲突
- 1 个秘密如果被揭露，会让某个 NPC 的现有生活崩塌
- 1 处道德灰色地带（没有纯粹的好人坏人）
```

## 模块二：双重检索（每回合执行）

### 主角视角检索

直接用 Graphiti `search()`，以"主角"为 center_node，查相关人、地、Topic。返回的是主角"知道"的明面信息，用于叙述。

### 上帝视角检索

要的不是"图里所有 Topic"，而是"**和当前场景潜在相关、可能被触发的 Topic**"。

**三路并行查询：**

```
路径 A：实体关联查询（图遍历，确定性）
  起点：动作涉及的所有 Person/Location 节点
  遍历：1-2 跳内的 Topic 节点
  → "显然相关"的 Topic

路径 B：地点辐射查询（图遍历）
  起点：当前 Location 节点
  遍历：在此地发生过的事件、此地常驻 NPC 关联的 Topic
  → "环境氛围层面相关"的 Topic

路径 C：语义检索（Graphiti search）
  query：玩家动作的自然语言描述
  过滤：只要 Topic 类节点
  → "语义相关"的 Topic
```

三路并行的原因：只用 C 会漏掉"图上明明连着但语义不近"的秘密；只用 A 会漏掉"语义相关但图上没直连"的潜在线索。

### 排序与截取

查出来可能 8-10 个 Topic，全塞 prompt 不现实。打分取 top 3-5。

**Demo 阶段简单打分：**

| 因素 | 思路 |
|------|------|
| 主角已知吗？ | 已知 Topic 降权（已揭露过的不优先再触发） |
| 上次出现在叙述里多久前？ | 越久没出现越优先 |
| `topic_type` | secret > rumor > event > item |

打分函数随便写个加权和，**不要在调参上陷进去**。

### Graphiti search 接口的坑

`search()` 默认返回 `EntityEdge` 列表（边，不是节点），不区分 Person/Location/Topic。要么在结果里过滤，要么用 `_search()` 带 `node_labels=["Topic"]` 约束。

**这是跑 demo 时第一个要踩的坑**——`search()` 接口的参数能不能精细控制，决定了上帝视角检索能不能优雅实现。如果不能，就绕开 Graphiti 的封装直接 Cypher。

## 模块三：叙述生成 Prompt

### 写法演进的三个版本

**版本一（朴素塞法，会翻车）：** 把"主角已知信息"和"隐藏信息"放一起，让模型"不要直接说出隐藏信息"。结果：模型分不清，要么直接说出，要么过度保守完全不暗示。

**版本二（分层 + 决策框架，可用）：** 视觉上分开两段，强制模型先做"是否揭露"的决策再写叙述。

**版本三（加揭露通道约束，推荐起手）：** 在版本二基础上，限定"如果揭露，必须通过指定通道"。给 LLM 一个具体的叙事工具箱，避免"暗示"沦为悬浮的眨眼提示。

### 推荐 Prompt 模板（版本三）

````
# 角色
你是一个文字冒险游戏的叙述者，负责生成自然、真实的事件文本。

# 当前场景
地点：[当前地点]
在场人物：[列表]
游戏时间：[第 N 天 时段]

# 主角的视角（这些信息主角已知，可在叙述中自然体现）
- [从主角视角检索得到的信息列表]

# 剧情燃料（这些是隐藏信息，主角不知道）

[Topic-A] [描述]
  - 知情者：[列表]
  - 与主角的关联：[关系]
  - 上次在叙述中出现：[轮数 / 从未]

[Topic-B] ...
[Topic-C] ...

# 玩家动作
[动词-对象-修饰]

# 你的任务

第一步：判断本回合是否揭露任何隐藏信息。
  - 揭露条件：主角的动作与某 Topic 高度相关，且场景合理
  - 揭露程度：完全揭露 / 部分暗示 / 仅留线索 / 不涉及
  - 多数情况下应选"部分暗示"或"不涉及"，避免剧情过快推进

第二步：生成 100-200 字叙述
  - 真实的对话和行为，不是说明文
  - 如果选择揭露，让信息通过 NPC 的反应、环境细节、口误等方式自然透出
  - 如果选择不涉及，写出"什么都没发生"的真实感（NPC 敷衍、转移话题、沉默）

# 揭露的实现通道（如果选择揭露，必须通过以下通道之一）

通道 1：NPC 自己的反应失控
  例：问到敏感话题时手抖、脸色变、突然沉默
  适用：reveal_level = "hint" 或 "trace"

通道 2：环境物证
  例：桌上的催债字条、墙角的典当票据
  适用：reveal_level = "hint"
  限制：物证必须主角合理能看到

通道 3：第三方在场
  例：相关人物正好上门
  适用：reveal_level = "full" 或 "hint"
  限制：本回合在场人物里要有第三方，否则不可用

通道 4：主角的推理（基于已知信息）
  例：主角注意到 NPC 最近瘦了、铺子生意不差但他焦虑
  适用：reveal_level = "trace"
  限制：必须基于"主角视角"段落里已有的信息推出

不允许的揭露方式：
- NPC 主动倾诉秘密（除非有强烈剧情理由）
- 旁白上帝视角直接说明
- 主角的"直觉"（除非已知信息支撑）

# 命名一致性（重要）
叙述中所有 NPC 必须使用其全名，不使用别名、绰号或职业称谓。
对话中 NPC 称呼可以自然，但叙述部分（非引号内）必须用全名。

# 输出格式（JSON）

{
  "decision": {
    "topic_id": "Topic-A" | null,
    "reveal_level": "full" | "hint" | "trace" | "none",
    "reasoning": "一句话说明为什么这样处理"
  },
  "narrative": "100-200 字的叙述文本"
}
````

### 设计要点

**为什么强制 JSON 输出 + reasoning 字段：** 让模型先思考再写作，对抗 LLM 的"展示欲"。`reasoning` 在调试时让你看到模型在想什么。

**为什么有揭露通道：** LLM 写"暗示"翻车的本质是它没有"暗示需要载体"的概念。逼它选具体通道（NPC 反应 / 环境物证 / 第三方 / 主角推理），叙述就具体了。

**命名一致性约束的作用：** 叙述质量直接决定 Graphiti 抽出来的图谱质量。叙述里"铁匠"被称为"老张"，图谱就可能多建一个节点。这条规则丑陋但有效。

**关于 reasoning 是否跨回合记忆：** Demo 阶段每回合独立决策，不带历史 reasoning。容易让模型陷入自圆其说越绕越远。如果发现剧情跳跃感强，再考虑塞最近 1-2 回合的 reasoning 进去。

## 验证清单

按风险从高到低，跑 demo 时重点观察：

### 1. 图谱质量（最高风险）
- 跑 5-10 回合后，Neo4j Browser 打开看图，比读日志直观一万倍
- 数 EntityNode 数量：5 个 NPC 有没有变成 8 个节点（去重失败）
- 看 EntityEdge 是否有意义，还是抽得碎乱

### 2. 时序价值
- 看 EntityEdge 的 `valid_at` / `invalid_at` 分布
- 跑到 8-10 回合，如果没有任何 `invalid_at` 被设置，要么剧情太平淡，要么冲突检测没触发
- 关键测试："铁匠以前信任主角，后来不信任了" 这种演变能否在新事件中被 AI 利用

### 3. 检索可用性
- 检索回的文本片段，LLM 是真的用上了，还是其实没参考
- 测试方法：故意在第 N 回合查询时把检索结果替换成无关内容，看叙述是否变差

### 4. 揭露机制
- 初始化时设定的秘密 Topic，能否在合适时机被自然揭露
- 揭露后，主角的 `KNOWS_ABOUT` 边是否被 Graphiti 自动建出
- 这是 Topic 独立节点设计 + 双重检索 + 揭露通道这一整套设计的核心验证点

## 实施优先级

**第一阶段：跑通最小链路**
1. 写初始化 prompt，生成 JSON
2. JSON 转 episodes 喂给 Graphiti
3. 写一个最简化的回合循环（只用主角视角检索，不做上帝视角，不做揭露机制）
4. 跑 5-10 回合，看 Neo4j Browser 里的图

这一步主要验证：Graphiti 的 `add_episode` 抽取质量、去重、时序边是否正常工作。

**第二阶段：加上帝视角和揭露机制**
1. 实现三路并行的上帝视角检索
2. 升级叙述生成 prompt 到版本三（揭露通道）
3. 跑 10 回合，重点看揭露机制是否工作

**第三阶段：观察和调优**
- 根据验证清单的四项，定位主要问题
- 决定是否升级边类型、是否引入动态菜单、是否扩展 NPC 规模
